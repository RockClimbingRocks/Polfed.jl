# Function Documentation Practices

This page defines the documentation standard for this codebase.

It combines general API-doc practice with Julia-specific guidance and
`Documenter.jl` conventions.

## Baseline Principles

1. Document the function, not only one method, unless method behavior differs.
2. State mutation clearly (`!` functions must say what they mutate).
3. Document positional arguments, keyword arguments, and return values explicitly.
4. Describe units/scales/domains for numerical inputs (e.g., rescaled `[-1, 1]`).
5. Provide at least one runnable example for public APIs.
6. Add “See also” links for nearby functionality and related structs.

## Julia-Specific Practices

### 1) Put docstrings immediately before definitions

Julia’s documentation system attaches docstrings to the next expression.
No blank line or comment should appear between a docstring and target definition.

### 2) Start with a signature block

Start docstrings with one or more representative signatures (including keyword
placeholders when there are many keywords).

### 3) Use canonical sections

For consistency, use these headings when applicable:

- `# Arguments`
- `# Keyword Arguments`
- `# Returns`
- `# Throws`
- `# Examples`
- `# See also`

### 4) Prefer generic dispatch-oriented wording

Describe accepted abstract interfaces (`AbstractVector`, `AbstractMatrix`,
callable `f!(Y, X)`), then mention notable specialized behavior per method.

### 5) Mark mutating semantics precisely

If a function mutates only some inputs, specify exactly which ones.
Example: “Mutates `Y`; does not mutate `X`.”

### 6) Use doctestable examples where possible

Prefer `jldoctest` for examples that can run in docs CI. Use fenced `julia`
blocks for non-deterministic or heavy examples.

### 7) Cross-reference related API

Use `@ref` links for related functions/types so users can navigate the docs
graph quickly.

## Documenter.jl Practices

1. Use `@docs` for hand-picked public APIs.
2. Use `@autodocs` for broad module sweeps only when object ordering is clear.
3. Keep one canonical location per docstring to avoid duplicate-splice errors.
4. Use explicit module qualification when names are ambiguous.
5. Keep pages task-oriented: public API, internals, and reporting in separate pages.

## Package-Specific Conventions For Polfed

- Always state whether values are in original energy scale or rescaled scale.
- For callbacks, use exact call contracts in docs, e.g. `f!(Y, X)`.
- For plans/configs, document which fields are user inputs vs resolved values.
- For kernels/optimizations, document assumptions (real-only GPU paths,
  expected sparsity/value grouping).
- For report structs, document whether a field is measured, inferred, or estimated.

## Recommended Function Docstring Template

```julia
"""
    function_name(arg1, arg2; kw1=..., kw2=...) -> ReturnType

One-line summary in imperative mood.

# Arguments
- `arg1::Type`: Meaning and accepted range/shape.
- `arg2::Type`: Meaning and accepted range/shape.

# Keyword Arguments
- `kw1`: Purpose and default behavior.
- `kw2`: Purpose and default behavior.

# Returns
- `value`: Description of return contents and invariants.

# Throws
- `ArgumentError`: Trigger conditions.

# Examples
```jldoctest
julia> out = function_name(a, b; kw1=true)
...
```

# See also
[`related_function`](@ref), [`RelatedType`](@ref)
"""
```
